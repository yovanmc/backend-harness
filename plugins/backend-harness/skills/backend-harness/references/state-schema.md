# State Schema Reference

## Section 1: Why Disk State

A harness run is a potentially long-running orchestration — from context parsing through implementation, evaluation, fix iterations, and escalation — spanning many steps and potentially many Claude Agent turns. All state lives in memory during execution.

When a Claude Code session hits the context window limit, **context compaction** occurs: the engine summarizes and prunes the transcript, reducing accumulated context to fit the new window. This is a feature for general conversation, but for a harness run it is a **catastrophic loss of state** — everything in memory (current phase, which tasks have been completed, evaluation results, failure signatures) vanishes.

Disk persistence is the difference between a prototype and production infrastructure. After every labeled step — each task transition, each evaluation run, each escalation decision — the orchestrator writes the full state to disk. On context compaction or crash, when the orchestrator resumes, it reads `harness-state.json`, inspects the `phase` field, and continues from exactly where it left off. This makes the harness **resumable and compaction-resilient**.

Without disk state, a mid-run compaction forces the human to restart from scratch. With it, the run survives context churn seamlessly.

## Section 2: Location

The state file is:

```
plans/harness-state.json
```

in the **target repository** (the repo being worked on by the orchestrator), not in a `.beads/` directory or elsewhere. It is a sibling to:

- **`plans/context-brief.md`** — the conditional codebase-orientation brief (generated by `/harness-brainstorm` if the codebase is non-trivial; omitted if greenfield)
- **`plans/<date>-<topic>-plan.md`** — the implementation spec/plan from `/harness-brainstorm` (e.g., `plans/2026-05-30-auth-refactor-plan.md`)

**Important:** The `plans/` directory should be **gitignored** in the target repository. It contains ephemeral run state — UUIDs, timestamps, failure signatures, iteration counters — not source code. Every harness run creates a new `runId` and writes fresh state; old `plans/` artifacts are not committed and do not pollute the repo history.

## Section 3: Full Schema

Below is the canonical JSON schema for `harness-state.json`:

```json
{
  "runId": "uuid",
  "phase": "brief | implement | evaluate | fix | escalated | done",
  "createdAt": "iso8601",
  "updatedAt": "iso8601",
  "worktree": "/abs/path/to/worktree",
  "briefSkipped": false,
  "contextBriefPath": "plans/context-brief.md",
  "planPath": "plans/2026-05-30-topic-plan.md",
  "tasks": [
    { "id": "t1", "title": "...", "status": "pending|implementing|reviewing|evaluating|passed|failed", "commitSha": "..." }
  ],
  "evaluation": {
    "lastStrategy": "full | component-scoped",
    "components": { "PaymentService": { "status": "pass|fail", "lastRun": "iso8601" } }
  },
  "failureHistory": [
    { "iteration": 1, "signature": "xunit:PaymentTests.Refund_NegativeAmount", "component": "PaymentService" }
  ],
  "iterations": { "PaymentService": 2 },
  "escalation": null
}
```

## Section 4: Field Semantics

| Field | Type | Description |
|-------|------|-------------|
| `runId` | string (UUID) | A new UUID generated at the start of each harness run. Used to scope failure history to the current run only; failures from previous runs are not carried forward. Uniquely identifies one end-to-end harness session. |
| `phase` | enum string | The resume entry point. Tells the orchestrator where to continue after context compaction or crash. Values: `brief` (context parsing in progress), `implement` (task implementation in progress), `evaluate` (test/validation in progress), `fix` (fix iteration in progress), `escalated` (human escalation reached), `done` (run completed successfully). |
| `createdAt` | ISO 8601 timestamp | The time the run began (generated once, never updated). |
| `updatedAt` | ISO 8601 timestamp | The time the state was last written. Refreshed on every write to track recency. |
| `worktree` | absolute file path | The full path to the isolated git worktree created for this run. All work happens here; the orchestrator can safely experiment without touching the main branch. |
| `briefSkipped` | boolean | `true` if the orchestrator determined the target codebase was greenfield or trivial and skipped the Context Brief generation phase; `false` if a brief was generated. Used by the orchestrator to decide whether to re-read `contextBriefPath`. |
| `contextBriefPath` | relative file path | Path (relative to the target repo root) to the generated context brief. Omitted or null if `briefSkipped` is true. Example: `plans/context-brief.md`. |
| `planPath` | relative file path | Path (relative to the target repo root) to the implementation plan/spec generated by `/harness-brainstorm`. Example: `plans/2026-05-30-auth-refactor-plan.md`. |
| `tasks` | array of objects | List of top-level tasks defined in the plan. Each task tracks its own lifecycle from pending to passed/failed. |
| `tasks[].id` | string | Short identifier for the task, e.g. `t1`, `t2`, `backend_api_v2`. Used internally to refer to the task. |
| `tasks[].title` | string | Human-readable title of the task, e.g. "Implement payment endpoint". |
| `tasks[].status` | enum string | Current status of the task. Values: `pending` (not started), `implementing` (in progress), `reviewing` (waiting for code review), `evaluating` (under test), `passed` (completed and validated), `failed` (did not meet acceptance criteria). |
| `tasks[].commitSha` | string | The git commit SHA after this task completed its inner loop (one or more fix iterations). Allows the orchestrator to diff and understand what changed. |
| `evaluation.lastStrategy` | enum string | The strategy used in the most recent evaluation run. Values: `full` (run all tests, all components) or `component-scoped` (run only tests for the component being fixed). Drives the graduated re-evaluation logic (see `graduated-reevaluation.md`). |
| `evaluation.components` | object | Per-component evaluation results from the most recent run. Keys are component names (e.g. `PaymentService`, `AuthMiddleware`). Values are objects with `status` (pass/fail) and `lastRun` (ISO 8601 timestamp of when this component was last evaluated). |
| `failureHistory` | array of objects | **The oscillation substrate** — an append-only record of every failure signature encountered in this run. Each entry captures: when a test failed, what it was, which component it belongs to, and which fix iteration it occurred in. When a signature that was previously resolved reappears, oscillation is detected and triggers escalation. |
| `failureHistory[].iteration` | integer | The fix attempt number when this signature was first encountered (e.g., iteration 1, 2, or 3 for a component). |
| `failureHistory[].signature` | string | A stable, queryable identifier for the failure (e.g., `xunit:PaymentTests.Refund_NegativeAmount`, `jest:AuthService.test.js#line42`, `pytest:TestPayment::test_refund_negative`). Format is `<framework>:<test-identifier>` to allow cross-framework searches. |
| `failureHistory[].component` | string | The component that contains or exercises this failure. Used to attribute oscillation to a specific component and enforce the 3-iteration cap. |
| `iterations` | object | Per-component fix attempt counter. Keys are component names; values are the current iteration count for that component (1, 2, or 3). When a component reaches 3 iterations, the orchestrator escalates to the human (cap exceeded). |
| `escalation` | null or object | `null` until escalation occurs. When escalated, contains: `reason` (string: "oscillation" or "cap_exceeded"), `detail` (human-readable explanation), and `signatures` (array of signature strings that triggered the escalation). |

## Section 5: Write Discipline

To ensure state durability and prevent corruption from mid-write crashes:

1. **Before every state write**, set `updatedAt` to the current ISO 8601 timestamp (e.g., `2026-05-30T14:35:22.123Z`).

2. **Write to a temporary file first:**
   ```
   plans/harness-state.tmp.json
   ```
   This isolates the write operation.

3. **Atomically rename** the temp file to the canonical name:
   ```bash
   mv plans/harness-state.tmp.json plans/harness-state.json
   ```
   On POSIX systems (Linux, macOS), `mv` is atomic at the filesystem level, preventing partial-write corruption if the process crashes or the session ends mid-write.

4. **Every write is idempotent**: the state object is always complete (not a diff or patch). Rewriting the full object on each step is simple and correct.

This pattern ensures that `harness-state.json` is always in a valid, parseable state — either the old version (if the write was interrupted) or the new version (if the write completed). There is no intermediate corrupt state.
